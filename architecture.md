# 架构设计

> 本文档是 dws-pipeline-analyzer 演进的顶层设计依据。所有代码变更应与本架构对齐。
>
> 状态：2026-07 确立。三层架构已落地，关联影响分析单资产 MVP 已实现（见 §6.2）。
> 跨资产影响分析设计已定稿（见 §6.2.9，血缘平台加工层路线），待定义交互格式后启动。

---

## 1. 背景：为什么要重新设计架构

### 现状

dws-pipeline-analyzer 最初是一个"制品包文档化工具"——读 execution_tasks.xlsx，生成字段映射、资产说明书、技术设计文档。在这个过程中，它沉淀了一个相当完整的**SQL 理解引擎**（解析/拓扑/字段血缘/物理来源穿透）。

但这个引擎被"文档化任务"包裹着，没有作为独立能力暴露。随着业务需求扩展到**关联影响分析**等新任务，现有架构暴露了三个问题：

| 问题 | 表现 |
|------|------|
| 引擎被任务绑死 | 单条路径和批量路径曾各写一套解析逻辑（Step 3~7），单条更新后批量未同步，导致批量产出缺数据块 |
| 全貌信息缺失 | knowledge 只回答"怎么加工"，不回答"资产是什么"；表定义(DDL)做过解析但没利用起来；调度配置/归属信息散落在 raw 里用完即弃 |
| 输入单一 | 只有 xlsx 一种输入，无法承载表定义、业务背景、变更清单等多源输入 |

### 目标

将理解引擎确立为**显式的能力底座**，使其能服务多个上层任务（文档化、字段检索、关联影响分析、未来任务），而非仅服务于文档化。

---

## 2. 核心设计原则

1. **引擎只懂单资产** — analyze_pipeline 接收一个规则组的数据，产出一份 knowledge。它不需要知道有"批量"这回事，也不关心结果会被哪个任务消费。

2. **所有任务平等消费理解结果** — 文档化、字段检索、影响分析都是引擎之上的任务，任何一个任务的变更不影响其他任务。

3. **确定性解析与 AI 增强分离** — 血缘解析、反向匹配等是确定性逻辑，必须靠代码精确计算，不依赖 AI。AI 只用在非关键增强点（自然语言提取、措辞优化）。

4. **输入多源、各自独立加载** — 执行规则、表定义、业务背景、变更清单是不同类型的输入，各自独立加载、独立演进，不互相耦合。

5. **批量是编排层** — 单资产做好了，批量就是把循环加上。引擎本身不该为批量做特殊设计。

6. **文档同步是硬约束** — 大功能迭代合入时，必须同步更新 README.md 和 SKILL.md。文档落后于代码是债务，不是小事——新用户/协作者读到的文档如果不能反映真实能力，会误判工具边界、走弯路。合入前自检："README 的能力总览、输入说明、输出产物是否覆盖了这次改的东西？"

## 4. 理解引擎层（engine.py）详解

### 4.1 定位

理解引擎是整个系统的**单一真相**。所有上层任务对资产的"理解"都来自这里，不允许任何任务自己重新解析 SQL。

引擎是**纯函数**模块：无 print、不读 args、不写文件。进度输出和文件写入由调用方负责。

### 4.2 对外接口

| 函数 | 输入 | 输出 | 视角 |
|------|------|------|------|
| `analyze_pipeline()` | rules, target_fields, group_variables, dialect | (knowledge, parsed_map) | 过程：怎么加工 |
| `build_table_catalog()` | ddl_dir / ddl文本, 表名清单 | table_catalog | 实体：表结构 |
| `build_asset_profile()` | knowledge, raw, catalog | asset_profile | 实体：资产全貌 |
| `build_reverse_index()` | knowledge | {源表.字段 → 影响项} | 查询：反向血缘 |

### 4.3 knowledge 的定位（过程视角，保持不变）

knowledge 回答**"这个资产怎么加工数据"**：

```
knowledge = {
  meta:          目标表/规则数/方言/模式标签
  topology:      调度图/数据依赖/场景
  data_flow:     表/步骤/数据块/关联键追溯/结构化概述
  field_mappings: 字段映射/血缘/物理穿透
  quality:       质量问题
  business_logic: 步骤描述（自动生成）
  source:        源表/TargetFields/组变量
}
```

knowledge 的结构已经成熟（200+ 测试覆盖），**不往里塞实体信息**，避免破坏稳定性。实体维度的信息由 asset_profile 承担。

### 4.4 asset_profile 的定位（实体视角，新增）

asset_profile 回答**"这个资产是什么"**——所有任务的共同入口，关联影响分析的地基：

```
asset_profile = {
  identity: {            归属
    rule_group_code, rule_group_en,
    project_code, business_owner
  },
  schedule: {            调度
    exec_sequence 链,
    delete_mode / delete_condition,
    exchange_source_table（分区交换）,
    variables（平台变量）
  },
  structure: {           目标表完整结构
    target_table,
    fields: [{name, type, comment, nullable, ...}]（从DDL）
  },
  dependencies: {        依赖清单（从knowledge提炼）
    source_tables: [表名清单],
    source_fields: [{table, field, → 哪些目标字段}]（实体化依赖）
  }
}
```

**落地策略**：先在 engine.py 建骨架函数（从 knowledge + raw 提炼现有可得的字段），DDL 增强和完整结构后续填充。

## 5. 数据层（analyzer.py）详解

analyzer.py 瘦身为**数据读取 + CLI**：

| 职责 | 函数 |
|------|------|
| 读取制品包 Excel | `read_excel()` → rules / target_fields / group_variables / 平台配置 |
| CLI 入口 | `main()` — 组装参数、调 engine、写文件、输出进度 |
| 诊断输出 | Excel 格式异常时的诊断信息（现有逻辑保留） |
| 兼容层 | re-export engine 的符号（过渡期，保证现有 import 不破） |

### re-export 兼容策略

engine.py 抽离后，analyzer.py 末尾保留：

```python
# engine 已独立，以下 re-export 保证现有代码 `from analyzer import xxx` 不破。
# 新代码请直接 from engine import xxx。
from engine import (analyze_pipeline, build_topology, ParsedSQL, RawRule, ...)
```

现有 batch.py / field_search.py / 全部测试的 import 无需改动。稳定后逐步迁移到直接 import engine。

## 6. 任务层详解

### 6.1 资产文档化（已有）

```

流程: read_excel → analyze_pipeline → [AI增强] → view_generator

```

### 6.2 关联影响分析（MVP 已实现）

#### 6.2.1 范围与定位

**一期：单资产影响分析。** 跨资产（多资产图谱拼接）等单资产夯实后再扩展。

**工具定位（重要）：影响清单 + 定位器，不是权威影响报告。**
- 核心价值 = **自动化两端，把中间留给人**：
  - 确定无影响 → 机器替你挡掉，不用看
  - 确定有影响 → 机器替你圈出来，必须处理
  - 判不了 → 机器诚实地说不知道，需要人看代码
- "待确认"不是缺陷，是诚实边界。人最终还是要读代码，工具帮人快速定位影响点、过滤确定性内容。

#### 6.2.2 输入契约

**输入只描述"什么变了"，不声明"影响谁"。** 源端无知，本资产全知——源端发变更通知时不知道下游谁读了它，匹配全由我们做。这正是影响分析的核心价值。

```yaml
asset: 规则组编码或资产名          # 要分析的本资产
changes:                           # 一批变化，整批丢进来
  - table: ods_user_src
    field: user_id                 # 空 = 表级变化
    change_type: type_changed      # type_changed | field_removed | field_added | rename | table_dropped | schema_restructured
    before: varchar(20)
    after: varchar(50)
  - table: ods_old_src
    field: null                    # 表级变化
    change_type: table_dropped
```

#### 6.2.3 三层过滤

源端一批变更里大部分跟本资产无关。三层过滤分别处理：

| 过滤层 | 判定 | 处理 |
|--------|------|------|
| 表级：本资产没读 | 表名不在任何 step 的 source_tables | 丢弃，进附录 |
| 字段级：表读了但字段没用 | 表命中，字段不在该 step 的 source_fields | 标"表相关、字段无关"，进附录 |
| 命中 | 表和字段都用了 | 进入传播分析 |

没有前两层，会看到大量假阳性。

#### 6.2.4 表级变化处理

表级变化（field=null）按语义分两类，都给资产责任人最想要的东西——**"我哪里用了这张表"的完整触点清单**：

| 变化类型 | 处理 | 严重度 |
|----------|------|--------|
| **整表下线/删除** | 列出该表流向本资产的所有下游字段（爆炸半径），每个标"来源消失" | **高**（确定） |
| **表重构/不明确** | 同样列出所有触点，标"结构变了需逐项核对" | 待确认 |

#### 6.2.5 逐跳传播（核心算法）

传播是一次图遍历，图的"边"分两种：

```
跳内边（step 内）：  source_field →[表达式]→ output_alias
跳间边（step 间）：  step_N.output_alias  ===  step_{N+1}.source_field（按列名匹配）
```

**跳内边**：抽表达式里引用的列名（sqlglot 遍历 Column 节点），命中即传播。

| 子场景 | 处理 |
|--------|------|
| `cast(col_a as bigint)` 直接引用 | 抽列名，命中即传播 |
| `SELECT *` 通配 | **整条下游标"待确认"，不猜**；必须指出是哪个步骤用了 SELECT *，并说明"因通配无法枚举字段" |
| `col_a + col_b` 复合表达式 | 任一源变即传播，输出标"受影响" |
| 字段在源表但本步没引用 | 抽不到列名 → 自然不传播（正确） |

**跳间边**：中间表列名 = 上一步 SELECT 输出别名（INSERT/CTAS 几乎总成立），按名字匹配。MVP 假设中间表不重命名列；对不上则标"断链/待确认"，不强行连。

**铁律：传播与影响判定必须分离。**
- **传播**（确定性）：只回答"波及哪些目标字段"，纯图遍历。
- **影响判定**（带规则）：在波及字段上，结合变化类型 + 沿途表达式 + 目标 DDL 判严重度。

为什么分离？`cast(col as bigint)` 让 varchar 扩容**传播**到下游（值变了），但**影响**是 nil（类型被吸收）。不分离会把所有传播都报成"有影响"，假阳性爆炸。

```
变化 ──传播(确定性)──▶ 波及字段集合 ──影响判定(带规则)──▶ severity + reason
```

#### 6.2.6 输出：Excel 影响清单（MVP 主交付物）

**主交付物是 Excel，不是 HTML。** Excel 是"工作底稿"——可筛选、可排序、可批注、可 diff、可进 CI。HTML 作为可选展示层后做。

**输出主序 = 资产视角（以目标字段为一行）**，不是源变更视角。资产责任人拥有的是目标字段，输出要回答"我的哪些字段中招"。

**状态列是灵魂**（颜色从重到轻）：

```
🔴 有影响    传播到了 + 实质影响（类型破坏/字段消失/截断风险）
🟡 待确认    传播到了但判不了（SELECT */重命名/表重构断链）
🟢 无影响    传播到了但被吸收（cast 吸收/目标DDL兼容）
⚪ 未命中    没传播到 → 进附录，不进主表
```

**🟢无影响不进 Sheet1**（它不是"要做的事"），进 Sheet3 附录可追溯。Sheet1 只留需要人关注的（🔴🟡）。这是"清单思维"——不丢信息，但主表只放 actionable 的。

**多源字段处理**：一个目标字段被多个源变更命中时，**每个命中一行**，加"命中源数/总源数"列做分流。不综合判断合并语义——那是人读代码时自然判断的。同一字段多行是特性不是 bug。

**Sheet 结构**：

```
Sheet1: 影响清单（主表，只含 🔴🟡）
  状态 │ 目标表 │ 目标字段 │ 命中源数 │ 源表 │ 源字段 │ 变化类型 │ 去哪看(规则/步骤) │ 说明

Sheet2: 表级影响（整表下线专用）
  每行 = 本资产一个受波及字段，列: 源表(下线)、触点步骤、流向目标字段

Sheet3: 过滤摘要（🟢无影响 + ⚪未命中，可追溯不吵）
  每行 = 一个非主表的源变更，列: 源表、源字段、状态、原因
```

"去哪看"列是清单核心——人看完状态，下一个动作就是去这列指的规则+步骤读代码。为未来 AI 溯源预留。

#### 6.2.7 AI 演进路径（面向未来，现在铺地基）

**三层架构，每层只读上层输出，不越级：**

```
确定性层（现在做）：变更 → 传播 → 自包含清单      （AI 不碰，零幻觉）
解读层（未来挂）：  清单 → AI 逐条解读 → 报告      （AI 只读清单）
交互层（未来挂）：  人追问 → 按路径取子图 → AI      （应需取大JSON邻域）
```

**清单必须自包含**：每条原子事实带齐解读所需全部材料（源变更/传播路径+每跳表达式/目标DDL类型/机器判定状态+原因）。AI 读这一份就够，**不用回头翻资产大 JSON**。逐行解读，不碰全图——把幻觉面从"遍历整张图"压到"理解一行表达式"。

**不需要给 AI 做"读取器"压缩大 JSON**——那只减输入体积不减 AI 工作量（图遍历仍交给概率模型 = 幻觉）。真正减负的是**确定性传播层**，它替 AI 把最危险的活干掉。

**"按路径取子图"的读取器属于交互层**（人追问时按已算路径取邻域），不是核心分析的前置优化。

**AI 报告的天花板卡在输入质量上**：直接喂大 JSON 让 AI 推 = 幻觉重灾区。清单越干净、每条原子事实越规范，未来 AI 报告质量越高。现在清单做扎实 = 给未来 AI 铺确定性轨道。

**诚实边界**：AI 报告 ≠ 替代决策。能做到"高质量初判 + 显式置信度 + 可追溯证据"，做不到零幻觉/替代生产决策。每句 AI 结论都能点回清单某条原子事实——这条铁律现在就写进设计。

#### 6.2.8 落地步骤

| 步骤 | 内容 | 产出 | 状态 |
|------|------|------|------|
| 1 | 输入解析：读变更清单 Excel 三 Sheet → ChangeItem[] | 变更列表 | ✅ |
| 2 | 三层过滤：表级命中（平切短路）→ 字段级命中 → 丢弃 | 命中变更 | ✅ |
| 3 | 逐跳传播：沿 field_mappings 抽列依赖、跨跳别名对齐 | 波及目标字段集 | ✅ |
| 4 | 影响判定：类型兼容性链 + 变化类型映射表 → severity + reason | 影响清单 | ✅ |
| 5 | Excel 渲染：四 Sheet（统计摘要/影响清单/表级影响/过滤摘要） | impact.xlsx | ✅ |
| 6 | command：impact-analysis.md + 用户模板 | 可调用入口 | ✅ |
| 7（未来）| AI 解读层：清单 → 报告 | 影响报告 | 待 AI 增强 |

#### 6.2.9 跨资产影响分析（设计已定稿，待启动）

**定位：血缘平台加工层，不重新建图。**

已有血缘平台能做跨资产的表级/字段级影响分析，但输出是"压平"的原始清单——
只告诉你"这些表和字段受影响了"，缺层级结构、缺严重度、缺传播路径、缺兼容性判断。
我们的定位是做平台输出的**下游加工层**：平台管跨资产图遍历，我们管单资产深度分析。

**核心机制：逐层交替确认（非一次性全量加工）。**

平台给的间接影响里有很多假阳性（平台只知道"间接用了"，不知道到底有没有真影响）。
逐层交替确认——平台查一层直接影响，我们确认一层真实影响，只有确认有影响的才往下一层传：

```
第0层：源变更（ods.user_src.user_id 类型变了）

第1层：
  问平台 → 谁直接依赖 ods.user_src？→ 平台答：DWB_USER_F
  我们分析 DWB_USER_F → user_id 变化 → 目标 uid 受影响(🔴) → 确认传播

第2层：
  问平台 → 谁直接依赖 DWB_USER_F？→ 平台答：DWB_ORDER_F, DWB_TRADE_F
  我们分析 DWB_ORDER_F → uid → total_amount 不受影响(🟢) → 链路断
  我们分析 DWB_TRADE_F → uid → trade_uid 受影响(🔴) → 确认传播

第3层：... 直到某层全被过滤或达深度上限
```

**传播截断规则**（每层判定后决定是否传给下游）：
- 🔴有影响 → 传播（目标字段的变化传给下一层）
- 🟡待确认 → **也传播**（不确定不等于没影响，保守传下去，否则可能漏掉真实影响）
- 🟢无影响 → 不传播（确定无影响，链路断）
- ⚪未命中 → 不传播（字段没被引用）

**为什么逐层交替而非一次性全量**：
- 平台间接影响有大量假阳性，逐层过滤让链路越往后越收敛
- 每层有明确层级，能输出"第N层影响"
- 范围可控——不会一次爆炸到全量资产

**不建图、不全局搜索**：跨资产的图遍历平台已做，我们只做单资产深度加工。
原规划的 build_asset_profile / build_reverse_index 在此路线下不需要（消费者消失）。

**终止条件**：某层所有影响都被过滤（🟢⚪），或达深度上限（防自引用无限循环，如5层）。

**前置条件（落地第一步）：定义跨层交互格式。**

逐层交替需要两份格式契约（格式我们定，平台按此实现）：
1. **问平台的输入**：本层确认受影响的目标表+字段 → 平台返回下游直接依赖
2. **平台返回的输出**：下游资产清单 + 受影响字段（压平的，不带层级/严重度）

这两份格式定不下来，跨资产就卡住——是落地的第一道前置条件。
定义时需考虑：批量查询（一次传多个受影响字段，平台批量返回）、字段名对齐口径、
资产定位方式（规则组英文名→代码仓目录）。

### 6.3 字段检索（已有，无需改动）

field_search 已在复用 engine 的解析能力，保持现状。

## 10. 设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| engine 边界 | 模块级（engine.py），含数据类 | 数据类是引擎领域模型，应随引擎走 |
| 现有 import | re-export 兼容层 | 零回归，过渡期安全 |
| knowledge 定位 | 过程视角，不膨胀 | 结构已成熟，实体信息由 asset_profile 承担 |
| 全貌结构 | asset_profile 独立于 knowledge | 过程视角与实体视角分离，互不污染 |
| 业务背景 | 不进引擎 | 语义增强，非确定性解析产物 |
| 批量 | 编排层，一期不做 | 单资产做好后水到渠成 |
| 影响分析一期 | 单资产，不做跨资产 | 单资产夯实后再扩图谱，跨资产别名对齐/命名漂移太难 |
| 影响分析输入 | 只描述"什么变了"，不声明"影响谁" | 源端无知、本资产全知，匹配全由我们做，这是工具的核心价值 |
| 影响分析工具定位 | 清单 + 定位器，非权威报告 | 自动化两端（确定有/无影响），中间（判不了）留给人，人最终要读代码 |
| 影响分析输出 | Excel 清单为主，HTML 后做 | Excel 可筛选/批注/diff/进CI，是工作底稿；状态列是灵魂 |
| 影响分析主序 | 资产视角（目标字段为一行） | 资产责任人拥有目标字段，输出回答"我的哪些字段中招" |
| 🟢无影响归属 | 进 Sheet3 附录，不进 Sheet1 | 主表只留 actionable（🔴🟡），🟢不是要做的事；不丢信息只换抽屉 |
| 多源字段 | 每个命中一行，不综合判断 | 综合合并语义交给读代码的人，工具只摆清"几件事" |
| 传播与判定分离 | 传播确定性，判定带规则 | cast 吸收型变化会传播但无影响，不分离则假阳性爆炸 |
| SELECT * 通配 | 标待确认不猜，指出步骤+原因 | 诚实边界，宁可让人确认也不假装传播成功 |
| AI 数据流转 | 不做大JSON读取器，做自包含清单 | 压缩只减体积不减工作量，确定性层替AI干掉图遍历才真减负 |
| AI 报告路径 | 清单→解读→报告，三层叠加 | 现在铺的清单是未来AI的确定性轨道，叠加而非重写 |
| 目标表逻辑 | 统一 max(exec_sequence) | 单条批量共用，消除歧义 |
| DDL 定位 | 尽力而为增强层 | 缺了能做(L1)，有了更准(L2/L3) |
| DDL 文件匹配 | 包含匹配+多扩展名 | 真实DDL文件名常有前缀/后缀/不同扩展名 |
| 表结构来源 | 一期SQL目录，预留DB/平台接入 | 易用性第一，零依赖 |
| yml 输入 | 和 xlsx 平等支持 | 代码仓数据另一种加载方式，产出完全一致 |
| DDL 注释格式 | 支持单引号+双引号+行内-- | 真实DDL导出工具格式不统一 |
| 类型变化判定 | 兼容性链（源新类型 vs 目标DDL + cast关口），纯看兼容性 | 不再"有cast就待确认"，能判的精确判；兼容就🟢不兼容就🔴 |
| 类型归一化 | nvarchar/character varying/varchar2 → varchar 家族 | 各种DDL导出写法统一，提取长度和精度标度 |
| asset_profile 时机 | 不建（跨资产走平台加工层路线） | 血缘平台管图遍历，我们管单资产深度分析；自建图的消费者消失 |
| SQL预处理 | 注释剔除在最早一步做一次 | 注释里的分号不能截断SQL，入口统一处理后下游拿到的都是干净SQL |
| 交叉测试矩阵 | SQL结构×解析路径系统性覆盖 | 防"修一处漏其他处"——同逻辑多处复制时矩阵能拦住 |
| 多规则组串联 | 统一靠数据依赖追溯（不靠表名尾缀） | 覆盖 mid/tmp/拆分规则组所有场景；尾缀只做加速 |
| 多规则组命令 | 单独 /analyze-chain（不混入 /analyze） | 产出和逻辑路径不同，不污染单资产的简洁性 |
| 多规则组追溯源 | 规则组穿透后的源表（不是单规则源表） | 一个规则组多步骤，要看整体读了哪些表 |

## 11. I 视图（已实现）+ LTS 调度（待启动）

### I 视图（资产是对外 I 视图，不是 F 表）✅ 已实现

**背景**：资产是 I 视图（dwb_xxx_i），F 表只是底表。现有分析只到 F 表，缺 F→I 这一段。
- 绝大多数 I 视图：直封（SELECT * FROM dwb_xxx_f），命名规则 _f→_i
- 少量 I 视图：有逻辑，命名可能不规律

**已实现**（discover_i_view + analyzer main Step 2b）：
- 发现策略：两步走
  - Step1 按名字找（_f→_i，覆盖95%直封，快路径）
  - Step2 名字没找到→全局搜索 view/ 目录里 FROM 引用 F 表的（兜底）
- 只有 _f 结尾的表才触发 I 视图发现（非 _f 表如 _d 不触发）
- I 视图作为最后一步追加到 rules（F→I），target_table 变成 I 视图
- DDL 发现返回 schema 目录（rglob 递归扫 table/ + view/）
- 找不到 I 视图→以 F 表为终点（容错，不阻塞）

**完整链路**：ODS → F表（加工） → I视图（封装） → 对外资产

## 12. 多规则组链路分析（待实施）

### 背景

一个资产可能由多个规则组协作加工完成：
- mid 表规则组：写 `_mid_f` 表（无 I 视图）
- 临时表规则组：写 `tmp_xxx`（无 I 视图）
- 最终 F 表规则组：读 mid/tmp 表，写最终 F 表（有 I 视图）

现有 `/analyze` 只分析单个规则组，看不到从 ODS 源表到最终资产的完整链路。

### 场景模型

```
规则组A → 写 dwb_trade_mid_f（mid表，无I视图）
规则组B → 写 dwb_detail_mid_f（mid表，无I视图）
规则组C → 读 mid表 → 写 dwb_trade_order_f（最终F表，有I视图）
```

特点：
- 每个规则组各自从 seq 1 开始（独立的）
- 连接点是"谁写的表被谁读了"——靠数据依赖，不靠表名尾缀
- 多对多：一个 mid 可能被多个 F 规则组读，一个 F 规则组可能读多个 mid
- 只有最终 F 表有 I 视图，mid/tmp 表没有

### 命令

单独命令 `/analyze-chain`（不走现有 /analyze 路径，产出和逻辑不同）。

```
/analyze-chain DWB_TRADE_ORDER_F          # 给最终F表规则组名
/analyze-chain @规则组目录                  # 给最终F表规则组目录
```

### 串联算法（统一靠数据依赖追溯）

```
输入：最终 F 表规则组

1. 定位规则组目录 → read_yml → analyze_pipeline 分析
2. 从 topology.steps 提取穿透后的所有源表（source_tables_from_sql）
   ★ 注意：是规则组穿透后的源表，不是单条规则的源表
3. 筛出 mid_f / tmp 表（这些是上游规则组的产出）
4. 在同子项目目录下，建索引找"谁写了这些表" → 上游规则组
5. 对每个上游规则组递归 Step 1-4
6. 直到源表都是 ods（没有规则组写它们）→ 到底了
7. 合并所有规则组的 rules，exec_sequence 拓扑排序（上游在前，下游在后）
8. 作为一个整体跑 analyze_pipeline → 产出一份完整链路报告
```

递归终止：某层规则组的穿透源表里没有 mid_f / tmp 表。

### "谁写了这张表"的索引

启动时扫一遍代码仓，建索引：`{target_table → [规则组目录]}`。
- 只读 yml 的 target_table 字段，不解析 SQL
- 预计 1-2 秒（几百个规则组），可缓存
- 后续追溯是 O(1) 字典查询

### exec_sequence 重新编号

合并后每个规则组各自从 seq 1 开始，需要全局重新编号：
- 按拓扑排序：被依赖的排前面，依赖别人的排后面
- 同层规则组（互相不依赖）的 seq 可以连续编号

### I 视图发现

只对最终 F 表触发 I 视图发现（mid/tmp 规则组不触发）。
合并后的 analyze_pipeline，I 视图发现逻辑检测最大 exec_sequence 的步骤
（最终 F 表），按现有逻辑发现 I 视图。

### 产出区分（跟单规则组不同）

- 报告标题标明"多规则组链路分析"
- 加"链路总览"section：展示规则组之间的关系图（谁写到哪、谁从哪读）
- 字段映射穿透 mid 表（从 ods 源表一路穿到最终 F 表字段）
- 数据流图更大（多个规则组的节点都画出来）
- 技术设计文档按规则组分段

### 待确认的细节

| # | 问题 | 当前草案 |
|---|------|---------|
| 1 | 规则组目录不在同子项目下怎么办 | 先在同子项目下找；找不到提示用户 --merge 手动指定 |
| 2 | 链路太深（5层mid）性能 | 建索引1-2秒 + 内存查询，总耗时10秒内（正常3-4层） |
| 3 | mid 表名不叫 _mid_f 怎么办 | 数据依赖追溯不依赖表名，能找到；但筛 mid 时用尾缀加速 |
| 4 | 拆分规则组（尾缀数字） | 同一个逻辑资产拆成多个规则组，靠数据依赖（tmp表串联）也能追到 |
